1. 프로젝트 개요

본 프로젝트의 목표는 PA2 명세에서 요구한 Unix-like shell과 여섯 개의 simplified GNU core utility 프로그램을 직접 구현하는 것이다. 구현 대상 core utility는 head, tail, cat, cp, mv, rm이며, 각각 pa2_head, pa2_tail, pa2_cat, pa2_cp, pa2_mv, pa2_rm이라는 실행 파일 이름을 갖도록 구성하였다. Shell 프로그램은 pa2라는 이름으로 빌드되며, 외부 executable 실행, self-implemented pa2_* executable 실행, builtin command 실행, redirection, pipeline, background job, foreground job, signal handling, script mode를 지원하도록 구현하였다.

과제의 핵심은 단순히 명령어별 기능을 흉내 내는 것이 아니라, Linux process model과 file descriptor model을 이해하고 이를 실제 shell 동작에 맞게 적용하는 데 있다. 따라서 구현 과정에서는 exec 계열 함수를 통한 프로그램 실행, fork를 통한 child process 생성, pipe를 통한 process 간 stream 연결, dup2를 통한 standard stream 교체, waitpid를 통한 child process 회수, setpgid를 통한 process group 분리, signal handler를 통한 Ctrl-C 및 Ctrl-Z 처리 등을 중심으로 설계하였다.

또한 PA2 명세에서는 system, popen, pclose 사용을 금지하고, 동적으로 할당된 resource를 프로그램 종료 전에 정리하도록 요구한다. 이에 따라 core utility는 외부 프로그램을 실행하지 않고 C/POSIX system call과 standard library를 이용해 직접 동작을 구현하였다. Shell 역시 외부 명령 실행은 execvp 또는 직접 구성한 fallback path를 통해 수행하되, shell 자체의 기능은 skeleton parser와 job control 구조를 기반으로 직접 처리하였다.


2. 전체 설계 방향

전체 구현은 “명세 준수”, “Unix-like 동작”, “작은 단위의 책임 분리”, “resource 정리”를 기준으로 진행하였다. PA2는 hidden test가 error message와 exit status까지 확인할 가능성이 높기 때문에, 기능이 동작하는 것뿐 아니라 출력 형식과 실패 시 status code도 중요하다. 따라서 각 command에서 성공과 실패 경로를 명확히 나누고, 실패 시에는 명세와 GNU coreutils의 일반적인 형식에 가까운 error message를 출력하도록 구성하였다.

Core utility 쪽은 각 실행 파일이 독립적으로 동작하도록 설계하였다. head와 tail은 선택적인 -n argument를 처리하고, file argument가 없거나 -로 주어진 경우 standard input을 읽는다. cat, cp, mv, rm은 file system operation을 직접 수행하며, file descriptor와 memory allocation이 필요한 경우 함수 내부에서 소유권을 분명히 하도록 구현하였다.

Shell 쪽은 parser가 만들어 주는 Input, Pipeline, Command 구조를 중심으로 실행 흐름을 구성하였다. 사용자가 입력한 한 줄은 하나 이상의 pipeline으로 분해되고, 각 pipeline은 command list와 background 여부를 가진다. Shell은 command type을 확인한 뒤 builtin command인지, pa2_* executable인지, 일반 executable인지에 따라 실행 경로를 선택한다. Pipeline에 포함된 command는 모두 별도 process에서 동시에 실행되도록 구성하였고, 단일 builtin command는 shell process에서 직접 실행하여 cd, exit처럼 shell 상태를 바꾸어야 하는 명령이 정상적으로 반영되게 하였다.

Shell이 생성한 child process는 shell과 다른 process group에 속하도록 처리하였다. 이는 PA2 명세에서 child process의 pgid가 shell의 pgid와 같으면 채점하지 않는다고 명시되어 있기 때문에 매우 중요한 조건이다. Foreground job은 terminal control을 job의 process group으로 넘겨 실행하고, 종료 또는 정지 후에는 다시 shell의 process group으로 terminal control을 되돌리도록 구성하였다.


3. 디렉터리 구조와 빌드 방식

제출 디렉터리는 PA2 skeleton 구조를 유지하도록 구성하였다. Makefile은 source file을 compile하여 object file은 build 디렉터리에, 최종 executable은 bin 디렉터리에 생성한다. 빌드 후 생성되는 bin과 build는 실행을 위해 필요한 중간 결과물이지만, 제출 archive에는 포함하지 않아야 하므로 제출 전 make clean을 수행하는 것을 기준으로 하였다.

주요 디렉터리 역할은 다음과 같다.

PA2/Makefile은 전체 build rule과 clean rule을 정의한다.

PA2/executables_src 디렉터리는 pa2_head, pa2_tail, pa2_cat, pa2_cp, pa2_mv, pa2_rm 등 core utility 구현을 포함한다.

PA2/shell_src 디렉터리는 pa2 shell, parser, helper, builtin command, job, jobs 관련 source file과 header file을 포함한다.

PA2/bin 디렉터리는 make 실행 후 생성되는 executable들이 위치하는 곳이며, 제출 전에는 제거되어야 한다.

PA2/build 디렉터리는 object file이 위치하는 곳이며, bin과 마찬가지로 제출 전에는 제거되어야 한다.

Makefile은 PA2 명세의 요구에 맞게 모든 executable을 한 번에 build할 수 있도록 구성되어 있다. 채점 환경에서는 제출 archive를 풀고 make를 실행하므로, 별도의 수동 compile 명령이나 외부 build script 없이 Makefile만으로 전체 프로그램이 생성되어야 한다. 이 점을 고려하여 source file dependency와 build output 경로를 Makefile 안에서 일관되게 관리하였다.


4. Core Utility 실행 파일 구현

4.1 pa2_head

pa2_head는 file 또는 standard input에서 처음 N개의 line을 출력하는 프로그램이다. -n option이 주어지지 않으면 기본값으로 10줄을 출력한다. Option parsing은 getopt를 사용하여 처리하였다. PA2 instruction에서는 getopt를 사용할 수 있다고 설명하고 있으며, head와 tail처럼 option이 있는 command에서는 직접 문자열을 분해하는 것보다 getopt를 사용하는 편이 더 안정적이다.

구현에서는 getopt(argc, argv, "n:") 형태를 사용하였다. 이를 통해 pa2_head -n 3 file뿐 아니라 pa2_head file -n 3과 같이 file name 뒤에 option이 오는 경우도 처리할 수 있다. PDF 예시와 hidden test에서 option 위치가 고정되어 있지 않을 수 있으므로, non-option 뒤의 option도 인식하는 방식이 필요하다고 판단하였다.

입력 file이 없거나 file argument가 -인 경우에는 standard input을 읽는다. File path가 주어진 경우에는 open 또는 fopen 전에 stat 결과와 open 실패 여부를 확인하여 file이 존재하지 않는 경우, directory를 읽으려는 경우, permission 문제가 있는 경우를 구분한다. Line count가 숫자가 아니거나 음수에 해당하는 경우에는 invalid number of lines error를 출력하고 실패로 종료한다.

4.2 pa2_tail

pa2_tail은 file 또는 standard input에서 마지막 N개의 line을 출력한다. 기본 line count는 head와 동일하게 10이다. tail은 전체 입력 중 마지막 N줄만 필요하므로, 모든 line을 무조건 저장하는 대신 circular buffer를 사용하여 memory 사용량을 줄였다.

각 line은 getline으로 읽고, circular buffer의 현재 slot에 저장한다. 이미 해당 slot에 이전 line이 저장되어 있으면 새 line을 저장하기 전에 기존 memory를 free한다. 입력이 끝난 뒤에는 실제 저장된 line 수와 circular buffer의 시작 위치를 계산하여 올바른 순서로 출력한다. 출력 후에는 buffer에 남아 있는 모든 line과 metadata array를 free한다.

Option 처리 방식은 pa2_head와 동일하게 getopt(argc, argv, "n:")를 사용하였다. 따라서 -n option이 command 앞쪽에 있든 file name 뒤쪽에 있든 같은 방식으로 처리된다. File argument가 생략되거나 -로 주어진 경우에는 standard input을 읽고, file이 존재하지 않거나 directory인 경우에는 명세에 맞는 error message를 출력한다.

4.3 pa2_cat

pa2_cat은 하나 이상의 file을 순서대로 읽어 standard output에 그대로 출력한다. File argument가 없으면 standard input을 읽으며, argument가 -인 경우에도 standard input을 입력 source로 사용한다. 여러 file이 주어진 경우에는 각 file을 순서대로 처리하고, file 사이에 임의의 공백이나 newline을 추가하지 않는다.

입출력은 read와 write를 사용하여 구현하였다. write는 요청한 byte 수보다 적은 byte만 기록할 수 있으므로, short write가 발생하더라도 남은 byte를 모두 출력할 때까지 반복하도록 처리하였다. 이 방식은 pipe나 redirected output처럼 write 대상이 일반 file이 아닐 때도 안정적으로 동작한다.

pa2_cat은 file open 실패를 errno 기준으로 처리한다. 존재하지 않는 file, directory 입력, permission error는 각각 적절한 message를 standard error로 출력한다. 여러 file 중 일부에서 error가 발생하더라도 가능한 나머지 file은 계속 처리하며, 하나라도 실패가 있었다면 전체 exit status는 failure가 되도록 구성하였다.

4.4 pa2_cp

pa2_cp는 source file을 destination으로 복사하는 프로그램이다. 하나의 source와 destination이 주어진 경우에는 destination이 기존 directory인지 file인지에 따라 target path를 결정한다. 여러 source가 주어진 경우에는 마지막 argument가 directory여야 하며, 각 source는 destination directory 아래의 basename으로 복사된다.

복사 동작은 외부 cp 프로그램을 실행하지 않고 직접 구현하였다. Source file은 read 전용으로 열고, target file은 create/truncate/write mode로 연다. 읽은 buffer는 write를 반복 호출하여 모두 기록한다. 이때 write가 partial write를 반환할 수 있으므로, 기록이 완료될 때까지 offset을 갱신하며 반복한다.

Destination이 directory인 경우 target path는 destination path와 basename(source)을 결합하여 만든다. 이 과정에서 동적으로 할당한 path 문자열은 복사 작업이 끝난 뒤 반드시 free한다. Source와 target이 같은 file인지 확인하기 위해 stat 결과의 device number와 inode number를 비교한다. 같은 file로 복사하려는 경우에는 파일 내용을 덮어써서 손상시키지 않도록 operation을 거부한다.

Trailing slash가 붙은 destination 처리도 중요하게 보았다. 예를 들어 destination이 bar/ 형태인데 실제 bar가 directory가 아니라면, 명세에서 기대하는 Not a directory 계열 error를 출력해야 한다. 이러한 edge case는 일반적인 성공 경로보다 hidden test에서 더 쉽게 드러나는 부분이므로 target resolution 단계에서 별도로 처리하였다.

4.5 pa2_mv

pa2_mv는 source를 destination으로 이동하거나 이름을 변경하는 프로그램이다. 실제 이동 작업은 rename system call을 중심으로 구현하였다. 하나의 source와 destination이 주어진 경우에는 destination이 directory인지 여부에 따라 최종 target path를 결정하고, 여러 source가 주어진 경우에는 destination이 반드시 directory여야 한다.

mv는 source file뿐 아니라 directory source도 다룰 수 있어야 한다. 따라서 source와 target의 stat 정보를 확인하고, 같은 file로 이동하려는 경우에는 operation을 거부한다. Directory를 자기 자신의 하위 directory로 이동하려는 경우도 file system 구조를 깨뜨릴 수 있으므로 realpath 기반 검사로 방지한다.

PA2 PDF에서는 mv error message에서 source를 출력할 때 basename만 사용하라는 조건이 있다. 이에 따라 cannot stat, missing destination operand, same file message 등 source를 보여 주는 error path에서는 전체 path가 아니라 pa2_basename(source)를 사용하도록 구성하였다. 예를 들어 /tmp/test/sub/foo를 source로 넘겼더라도 error message에는 foo만 표시되도록 하였다.

4.6 pa2_rm

pa2_rm은 file을 삭제하는 프로그램이다. 여러 file argument를 받을 수 있으며, 각 file에 대해 unlink를 수행한다. Directory는 삭제 대상이 아니므로 directory가 주어진 경우에는 Is a directory error를 출력하고 삭제하지 않는다.

File이 존재하지 않는 경우, permission이 없는 경우, directory가 주어진 경우를 errno와 stat 결과를 이용해 구분한다. 여러 file 중 일부가 실패하더라도 나머지 file은 계속 처리한다. 하나 이상의 실패가 발생한 경우 전체 exit status는 failure가 되도록 하여 일반 Unix command와 유사하게 동작하게 하였다.


5. Shell 실행 구조

Shell은 사용자 입력을 읽고 parser를 통해 구조화한 뒤, 각 pipeline과 command를 순서대로 실행한다. Skeleton에서 제공된 parser는 입력 문자열을 Input, Pipeline, Command 구조로 변환한다. Input은 한 줄 안에 포함된 여러 command line을 나타내고, Pipeline은 pipe로 연결된 command들의 묶음과 background 실행 여부를 가진다. Command는 argv, stdin redirection path, stdout redirection path, append 여부, command type, next command pointer를 가진다.

실행 흐름은 먼저 입력을 parsing하고, 각 pipeline을 순회하면서 실행하는 방식이다. Pipeline에 command가 하나이고 builtin command인 경우에는 가능한 한 shell process 안에서 직접 실행한다. cd는 현재 shell의 working directory를 변경해야 하고, exit는 shell process 자체를 종료해야 하므로 child process에서 실행하면 의미가 없다. 따라서 top-level 단일 builtin은 별도 path로 처리하였다.

반대로 pipeline 안에 들어간 builtin command는 일반 command와 마찬가지로 child process에서 실행한다. 예를 들어 echo 1 | exit | echo 2와 같은 pipeline에서 exit는 shell을 종료시키면 안 되고, pipeline의 한 subprocess만 종료시켜야 한다. 이 조건을 만족시키기 위해 pipeline 실행 경로에서는 builtin도 fork된 child 안에서 수행하게 하였다.

외부 command 실행은 execvp를 기본으로 한다. execvp는 PATH에 포함된 executable을 찾아 실행한다. 다만 PA2 명세에서는 pa2_* executable을 bin directory에 넣고, shell에서 pa2_cat file.txt처럼 path 없이 실행할 수 있어야 한다. 사용자가 항상 PATH에 PA2/bin을 추가한다고 가정하면 hidden test 환경에서 실패할 수 있으므로, execvp가 ENOENT로 실패한 경우 shell executable 위치를 기준으로 bin/command path를 만들어 한 번 더 실행을 시도하도록 하였다. 이를 통해 PA2/bin이 PATH에 없어도 pa2_* executable을 실행할 수 있다.

Command 실행이 끝나면 shell은 exit status를 저장한다. 일반 command가 존재하지 않는 경우에는 standard error에 command not found message를 출력하고 status code를 127로 설정한다. 정상 종료한 foreground command는 해당 exit status를 last status로 저장하고, signal로 종료된 경우에는 128 + signal number 형태를 사용한다.


6. Redirection 구현

Redirection은 command 실행 전에 standard input 또는 standard output을 file로 교체하는 기능이다. Input redirection은 < operator를 사용하며, 지정된 file을 read-only로 열어 STDIN_FILENO에 dup2한다. Output redirection은 > 또는 >> operator를 사용하며, >는 file을 새로 쓰거나 truncate하고, >>는 기존 file 끝에 append한다.

단일 builtin command는 shell process 안에서 실행될 수 있기 때문에 redirection 처리 시 주의가 필요하다. Shell process의 stdin/stdout을 직접 바꾸면 builtin 실행 후에도 shell의 standard stream이 바뀐 상태로 남을 수 있다. 이를 방지하기 위해 단일 builtin 실행 전에는 원래 stdin/stdout을 dup으로 백업하고, redirection을 적용한 뒤 builtin을 실행한다. 실행이 끝나면 백업해 둔 file descriptor를 이용해 원래 standard stream을 복구한다.

Fork된 child process에서 실행되는 command는 redirection을 적용한 뒤 exec 또는 builtin function을 수행한다. Child process는 곧 새 program으로 대체되거나 종료되므로, shell process의 standard stream에는 영향을 주지 않는다. Redirection file open에 실패한 경우에는 pa2_shell: pathname: strerror(errno) 형식으로 error를 출력하고 해당 command 실행을 실패 처리한다.

Pipeline과 redirection이 동시에 사용되는 경우에는 redirection이 pipe 연결과 충돌하지 않도록 command 위치를 고려한다. 첫 command의 stdin은 shell stdin 또는 input redirection에서 오고, 마지막 command의 stdout은 shell stdout 또는 output redirection으로 향한다. 중간 command는 pipe의 read end와 write end를 standard stream으로 사용한다.


7. Pipeline 구현

Pipeline은 여러 command를 pipe로 연결하여 동시에 실행하는 구조이다. PA2 PDF는 command를 순차적으로 실행하지 말고, pipeline에 포함된 process들을 모두 생성한 뒤 동시에 실행해야 한다고 설명한다. 따라서 구현에서는 pipeline command 수에 맞춰 필요한 pipe를 만들고, 각 command마다 fork를 수행한 뒤 적절한 pipe end를 stdin 또는 stdout에 연결하였다.

예를 들어 command1 | command2 | command3 형태라면 command1의 stdout은 첫 번째 pipe의 write end로 연결되고, command2의 stdin은 첫 번째 pipe의 read end로, command2의 stdout은 두 번째 pipe의 write end로 연결된다. command3의 stdin은 두 번째 pipe의 read end로 연결되고, command3의 stdout은 기본 stdout 또는 redirection target으로 향한다.

Pipe file descriptor는 parent와 child 모두에서 신중하게 닫아야 한다. 사용하지 않는 pipe end가 열린 채로 남아 있으면 reader process가 EOF를 받지 못해 종료하지 않는 문제가 생길 수 있다. 따라서 child에서는 자신의 stdin/stdout에 필요한 descriptor를 dup2한 뒤 모든 pipe descriptor를 닫고, parent에서도 child 생성 후 더 이상 필요 없는 pipe descriptor를 닫는다.

Pipeline 전체는 하나의 job으로 취급한다. 첫 번째 child process의 pid를 process group id로 사용하고, 이후 같은 pipeline에 속한 child process들은 같은 process group에 넣는다. Foreground pipeline이면 shell은 해당 job을 foreground로 넘기고 모든 child가 종료되거나 정지될 때까지 wait한다. Background pipeline이면 shell은 즉시 prompt로 돌아가되 job table에 job 정보를 저장한다.

Pipeline 안의 exit builtin은 shell을 종료시키지 않고 해당 subprocess만 종료시킨다. 이 부분은 PA2 PDF의 설명과 일치하도록 처리하였다. 즉 top-level 단일 exit command는 shell 종료 요청으로 처리하지만, pipeline 내부에서 실행되는 exit는 child process의 종료로만 해석한다.


8. Builtin Command 구현

Shell은 cd, pwd, exit, jobs, fg, bg builtin command를 지원한다. Builtin command는 외부 executable과 달리 shell 내부 상태를 직접 다루어야 하므로 별도의 dispatch 함수를 통해 처리하였다.

cd는 현재 working directory를 변경한다. Argument가 없으면 HOME directory로 이동하고, argument가 있으면 해당 path로 chdir을 수행한다. 이동 실패 시에는 errno에 따른 error message를 출력하고 failure status를 반환한다. cd는 shell process의 working directory를 바꾸어야 하므로 top-level 단일 command일 때 parent shell 안에서 직접 실행된다.

pwd는 현재 working directory를 출력한다. getcwd를 사용하여 현재 directory path를 얻고 standard output에 출력한다. Output redirection이 있는 경우에는 redirection이 적용된 stdout으로 출력된다.

exit는 shell 종료를 담당한다. Argument가 없으면 last command status를 종료 status로 사용하고, 숫자 argument가 있으면 해당 값을 status로 사용한다. Argument가 너무 많거나 숫자가 아닌 경우에는 error를 출력한다. Top-level exit는 active job이 있는 경우 즉시 shell을 종료하지 않고 There are still jobs active. message를 출력하여 사용자가 background job을 인지할 수 있게 한다. 반면 pipeline 내부에서 실행되는 exit는 child process만 종료하므로 active job 검사에 의해 status가 바뀌지 않도록 처리하였다.

jobs는 현재 job table에 등록된 job 목록을 출력한다. Background 또는 stopped job을 사용자에게 보여 주며, job id, process group, command 정보, 상태 정보를 포함한다. fg는 stopped 또는 background job을 foreground로 가져오고, bg는 stopped job을 background에서 계속 실행하도록 SIGCONT를 보낸다.

Builtin command dispatch 함수는 command 이름을 확인한 뒤 해당 구현 함수로 연결한다. exit의 경우에는 호출 결과가 shell 종료 여부와 관련되므로 return value를 정확히 상위 실행 흐름으로 전달해야 한다. 이 때문에 dispatch 단계에서 exit 처리 결과를 그대로 return하도록 하였다.


9. Job Control과 Signal 처리

PA2 shell은 foreground job과 background job을 구분하여 관리한다. Job은 하나의 process 또는 pipeline에 속한 여러 process들의 묶음이며, 같은 pipeline에 속한 process들은 같은 process group에 들어간다. Shell 자체는 별도의 process group에 남고, child process group과 shell process group이 같아지지 않도록 setpgid를 호출한다.

Foreground job을 실행할 때는 terminal control을 해당 job의 process group에 넘긴다. 사용자가 Ctrl-C를 누르면 SIGINT가 foreground process group으로 전달되어 foreground job이 종료될 수 있고, Ctrl-Z를 누르면 SIGTSTP가 전달되어 job이 정지될 수 있다. Foreground job이 종료되거나 정지된 뒤에는 terminal control을 다시 shell process group으로 되돌린다.

Background job은 terminal control을 받지 않고 실행된다. Shell은 background job을 job table에 등록하고 즉시 다음 입력을 받을 수 있도록 prompt로 돌아간다. 이후 waitpid를 WNOHANG 옵션 등과 함께 사용하여 종료된 background child를 정리하고 job 상태를 갱신한다.

Signal 처리는 shell과 child process의 역할을 구분하여 구성하였다. Interactive shell은 Ctrl-C나 Ctrl-Z를 받더라도 shell 자체가 종료되거나 정지되지 않아야 한다. 대신 foreground job이 있을 때 해당 signal이 job에 전달되도록 process group과 terminal control을 이용한다. Child process는 실행할 program의 일반적인 signal 동작을 따를 수 있도록 필요한 경우 default signal disposition을 사용한다.

Job table은 background 또는 stopped job을 추적하는 데 사용된다. Job이 완료되면 wait 결과를 반영하고, 더 이상 필요하지 않은 job 구조와 process 구조는 해제한다. Script mode나 EOF로 shell이 종료될 때 active job이 남아 있으면 child process도 resource라는 PA2 조건을 고려하여 남은 job을 정리하도록 하였다.


10. Script Mode 처리

pa2는 interactive mode뿐 아니라 script file을 argument로 받아 실행하는 mode도 지원한다. Script mode에서는 file에서 한 줄씩 command를 읽어 parser와 동일한 실행 경로로 전달한다. 이렇게 하면 interactive input과 script input이 같은 command execution logic을 공유하므로 behavior 차이를 줄일 수 있다.

Script mode에서 중요한 문제는 fork된 child process가 parent와 file offset을 공유할 수 있다는 점이다. 특히 pipeline 내부에서 exit builtin이 child process에서 libc exit를 호출하면, child가 상속한 script FILE stream 처리 과정에서 underlying file offset에 영향을 주어 parent가 같은 line을 다시 읽는 문제가 발생할 수 있다. 이를 방지하기 위해 script file은 open과 read를 이용해 전체 내용을 memory로 읽은 뒤 file descriptor를 닫고, 이후에는 memory buffer에서 line을 나누어 처리하도록 구성하였다.

이 방식은 child process가 더 이상 parent와 공유하는 script file descriptor를 건드리지 않게 하므로, pipeline 안의 builtin 종료가 script reading 위치를 바꾸는 문제를 제거한다. 또한 script 내용은 memory buffer로 관리되므로 실행이 끝난 뒤 한 번에 free할 수 있고, script file descriptor도 초기에 닫히므로 descriptor leak 가능성이 줄어든다.

Script 실행 중 EOF에 도달하거나 마지막 line 처리가 끝나면 shell은 남아 있는 background job을 확인하고 정리한 뒤 종료한다. 이는 interactive mode에서 사용자가 exit를 입력하지 않고 EOF를 보냈을 때도 동일하게 적용된다.


11. Resource Management

PA2 명세는 file, memory, child process를 resource로 보고, 동적으로 할당된 resource를 프로그램 종료 전에 정리해야 한다고 설명한다. 이를 반영하여 구현에서는 memory allocation, file descriptor, child process lifecycle을 각각 명확히 관리하려고 하였다.

Memory 측면에서는 parser가 만든 Input, Pipeline, Command 구조를 실행 후 free하고, job과 process 구조도 job 완료 후 해제한다. Core utility에서 동적으로 생성한 target path, getline으로 받은 line, circular buffer 내부 line pointer 등은 소유 주체가 끝나는 시점에 free한다. pa2_tail처럼 입력 크기에 따라 memory 사용량이 커질 수 있는 command는 circular buffer를 사용하여 필요한 line만 보관하도록 하였다.

File descriptor 측면에서는 open, pipe, dup으로 생성한 descriptor를 사용 후 close한다. Core utility에서는 input file descriptor와 output file descriptor를 복사 또는 출력 작업이 끝난 뒤 닫는다. Shell에서는 redirection 적용 후 원래 stdin/stdout을 복구하고, pipeline에서 사용하지 않는 pipe end를 parent와 child 양쪽에서 닫는다. 이러한 처리는 pipeline reader가 EOF를 정상적으로 받을 수 있게 하고, 장시간 shell을 실행했을 때 descriptor가 계속 쌓이는 문제를 줄인다.

Child process 측면에서는 foreground job은 waitpid를 통해 종료 또는 정지 상태를 회수한다. Background job은 job table에 등록해 두었다가 상태 변화가 있을 때 정리한다. Shell 종료 시점에는 남아 있는 background 또는 stopped job도 확인하여 child process가 orphan 상태로 방치되지 않도록 처리하였다.


12. Error Handling 정책

Error handling은 명세와 Unix-like convention을 함께 고려하여 구성하였다. File 관련 error는 errno를 기반으로 strerror(errno) 결과를 출력하고, command별 prefix를 붙였다. 예를 들어 pa2_cat은 pa2_cat: FILE: MESSAGE 형식을 사용하고, redirection 실패는 pa2_shell: pathname: MESSAGE 형식을 사용한다.

Command not found의 경우에는 EXECUTABLE: command not found를 standard error로 출력하고 status code를 127로 설정한다. 이는 PA2 Part 1 shell 설명에서 요구한 behavior이다. Builtin command의 argument error는 command별로 failure status를 반환하며, exit status는 shell의 last status 관리에 반영된다.

cp와 mv는 destination이 directory인지, source가 여러 개인지, trailing slash가 붙었는지에 따라 error message가 달라질 수 있다. 특히 mv는 PDF에서 source를 basename으로 출력하라고 요구하므로, source path 전체가 아니라 basename을 사용하도록 구현하였다. Same file 검사, cannot stat message, missing destination operand message도 이 규칙을 따르도록 하였다.

head와 tail은 -n argument가 잘못되었을 때 invalid number of lines message를 출력한다. File이 directory인 경우에는 단순 open 실패와 구분하여 error reading 계열 message를 출력한다. rm은 directory를 삭제하려는 경우 unlink를 시도해 실패 message를 그대로 내는 대신, directory임을 확인하여 Is a directory error를 출력한다.


13. 테스트 및 검증 내용

구현 후에는 make를 통해 전체 executable이 정상적으로 build되는지 확인하였다. Build 결과로 pa2, pa2_head, pa2_tail, pa2_cat, pa2_cp, pa2_mv, pa2_rm이 bin directory에 생성되는 것을 확인하고, 대표적인 정상 입력과 error 입력을 직접 실행하여 출력과 exit status를 점검하였다.

Core utility 쪽에서는 다음 범주의 동작을 확인하였다. head와 tail은 기본 10줄 출력, -n option 처리, file argument 생략 시 standard input 처리, file 뒤에 -n option이 오는 경우를 확인하였다. cat은 여러 file 연결, standard input 처리, 존재하지 않는 file error를 확인하였다. cp는 file 복사, directory destination 복사, multiple source 복사, same file 거부, trailing slash error를 확인하였다. mv는 file rename, directory destination 이동, missing source error, same file error, basename 출력 규칙을 확인하였다. rm은 file 삭제, 여러 file 처리, directory 거부를 확인하였다.

Shell 쪽에서는 외부 command 실행, pa2_* executable 실행, command not found status 127, input redirection, output redirection, append redirection, 단일 pipe, 여러 command pipeline, background job, jobs 출력, fg와 bg 동작, Ctrl-C와 Ctrl-Z 상황, builtin cd/pwd/exit 동작을 확인하였다. 또한 pipeline 안에서 exit가 실행될 때 shell 자체가 종료되지 않고 subprocess만 종료되는지도 점검하였다.

Script mode에서는 여러 command line을 순서대로 실행하는지, pipeline과 builtin이 섞인 line 이후 다음 line이 중복 실행되지 않는지, script EOF 도달 시 background child가 남지 않는지 확인하였다. 이 부분은 child process와 parent process가 file descriptor를 공유할 때 발생할 수 있는 문제를 방지하기 위한 중요한 검증 항목이었다.

제출 전에는 make clean을 수행하여 bin과 build directory가 archive에 포함되지 않도록 해야 한다. 최종 제출 archive는 source file과 Makefile만 포함해야 하며, 채점자는 archive를 풀고 make를 실행하여 build하므로 clean 상태에서도 Makefile 하나로 전체 executable이 다시 생성되어야 한다.


14. 결론

이번 PA2 구현에서는 Unix-like shell의 핵심 동작을 직접 구성하였다. Core utility 구현을 통해 file input/output, option parsing, file system operation, error handling을 연습하였고, shell 구현을 통해 process creation, process group, redirection, pipe, signal, job control, builtin command 처리 방식을 종합적으로 적용하였다.

특히 shell에서는 단일 command 실행과 pipeline 실행의 차이를 명확히 구분하는 것이 중요했다. cd와 top-level exit처럼 shell process의 상태를 직접 바꾸어야 하는 builtin은 parent shell에서 실행해야 하지만, pipeline 안의 builtin은 subprocess로 동작해야 한다. 또한 foreground job과 background job을 process group 단위로 관리해야 Ctrl-C, Ctrl-Z, jobs, fg, bg가 자연스럽게 동작한다.

전체적으로 구현은 PA2 PDF에서 요구한 기능과 제한 사항을 따르도록 작성하였다. system, popen, pclose를 사용하지 않고 직접 command behavior를 구현했으며, child process가 shell과 같은 process group에 남지 않도록 처리하였다. Redirection과 pipeline은 file descriptor 조작을 통해 구현하였고, script mode와 shell 종료 시에도 child process와 동적 resource를 정리하도록 구성하였다. 따라서 본 구현은 PA2가 요구하는 simplified core utility와 Unix-like shell의 주요 기능을 충족하는 것을 목표로 한다.
